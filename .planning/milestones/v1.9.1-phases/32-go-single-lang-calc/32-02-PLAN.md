## 1.1 Run /codepoint:scan

In the Go calculator project directory `tmp/go-calculator/`, trigger `/codepoint:scan`.

The scan skill executes a two-phase scan:
- Phase 1 (Overview): identifies modules, business areas, candidate flows
- Phase 2 (Deep Dive): after user confirmation, deep analysis and code point definition generation

## 1.2 Expected scan output structure

Scan should identify these business areas and flows:

**Collection**: calculator-related (name may vary — use structural matching)

**Flow 1**: API calculation flow
- Trigger: `POST /api/calculate`
- Entry: `internal/api/server.go` HandleCalculate function
- Path: calls `calculator.Evaluate()` through Parse->Validate->Compute->Format

**Flow 2**: Batch processing flow
- Trigger: CLI `--batch` or `-expr` mode
- Entry: `internal/batch/processor.go` ProcessExpressions function
- Path: calls `calculator.Evaluate()` through Parse->Validate->Compute->Format

**Flow 3**: History query flow
- Trigger: `GET /api/history/{id}`
- Entry: `internal/api/server.go` HandleHistoryGet function
- Path: calls `calculator.Evaluate()` to recompute stored expression through Parse->Validate->Compute->Format

**Shared code points** (across 3 flows):
- Parse function in `internal/calculator/calculator.go`
- Validate function in `internal/calculator/calculator.go`
- Compute function in `internal/calculator/calculator.go`
- Format function in `internal/calculator/calculator.go`

## 1.3 Verify .codepoints/ directory structure

After scan completes, verify these files exist and have valid content:

1. `tmp/go-calculator/.codepoints/index.json` — JSON index with collections, flows, points
2. `tmp/go-calculator/.codepoints/collections/` — at least 1 collection markdown file
3. `tmp/go-calculator/.codepoints/flows/` — at least 3 flow markdown files
4. `tmp/go-calculator/.codepoints/points/` — individual code point markdown files

## 1.4 Structural verification (NOT exact name matching)

Since scan is AI-driven and output naming may vary, verify **structure** not exact names:

**Minimum viable output thresholds:**

| Criterion | Minimum Threshold | Blocking |
|-----------|------------------|----------|
| Collections identified | >= 1 | Yes |
| Flows identified | >= 3 | Yes |
| Shared code points in >= 2 flows | >= 2 distinct shared points | Yes |
| Points with location (file:line) | >= 6 | Yes |
| Points with type (entry/boundary/state-change) | >= 4 | Yes |

**Structural checks:**
1. At least 2 flows reference code points in `internal/calculator/calculator.go` (shared pipeline)
2. At least 1 flow references code points in `internal/api/server.go` (API handler)
3. At least 1 flow references code points in `internal/batch/processor.go` (batch processor)
4. Code point IDs follow a naming convention (contain module/action identifiers, not arbitrary strings)

## 1.5 Scan correction procedure (if output is incomplete)

**If scan output fails any minimum threshold above**, apply the following correction steps:

1. **Identify the gap**: Which threshold failed? (too few flows? missing shared points?)
2. **Manual correction**: Edit the `.codepoints/` files directly to add missing structures:
   - Add missing flow files to `.codepoints/flows/` with proper trigger, sequence, and code point references
   - Add missing point files to `.codepoints/points/` with location, type, description
   - Update `index.json` to reference new files
3. **Validate correction**: Re-run the structural verification checks from 1.4
4. **Document what was corrected**: Note which items scan missed and what was manually added — this is a finding for FIX1 phase

**Correction is acceptable because:**
- Scan is AI-driven and non-deterministic (review concern)
- The goal is to verify the scan skill produces usable output, not that it's perfect on first run
- Manual correction simulates how a developer would fix scan issues in real usage
- Corrections are documented as skill deficiencies for FIX1 phase

## 1.6 Density validation

Check if the scan output or flow documentation mentions density/overlap metrics:

1. For each flow, count the number of code points in the sequence
2. Verify adjacent code points would have 20-60% stack overlap (based on their locations in the codebase)
3. If scan provides density analysis, verify it falls within the target range
4. If no density analysis provided, this is noted as a finding (not a blocking issue for SING-02)

## 1.7 Record findings for FIX1

Document any issues found with the scan skill (do NOT fix in this phase):
- Flow names don't follow expected convention
- Shared code points not identified
- Density analysis missing
- Incorrect locations (file:line off by more than 5 lines)
- Missing flows or collections
