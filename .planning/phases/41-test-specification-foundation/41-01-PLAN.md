**Title:** `# Code Point Test Plan -- Structured Frontend Test Flow Planning`

**Overview section:** Explain this skill generates test plans for frontend flows using a standardized 6-segment template. Include "When to Use" (after codepoint scan or plan, before implementation, for features with user interactions) and "When NOT to Use" (pure backend logic, one-off scripts, features with no user-facing behavior).

**Workflow section (6 steps, per D-02 interactive generation workflow):**

Step 1: Read Context
- If `.codepoints/index.json` exists, read it to extract flows, points, collections
- If no index.json, ask developer to describe the flow (name, trigger, stages)

Step 2: Define Test Scope
- Developer specifies which Flow to test (by flow_id from index.json, or describes a new one)
- Skill extracts preconditions from the flow definition (trigger, sequence of codepoints)

Step 3: Generate Test Cases (per D-04 three-segment format)
- For each stage in the flow's codepoint sequence, generate an Action -> Expected Response -> Verify triplet
- Cover three test case categories per data-model.md: normal (happy path), boundary (edge values, empty inputs), failure (invalid inputs, error states)
- Note: TC-{N} items should align with (but not duplicate) the `test_cases` field from the Flow definition in index.json. If index.json already has test_cases for this flow, reference them rather than regenerating.
- Each test case uses format:
```
### TC-{N}: {Title}
**Action:** {what the user does, behavioral description}
**Expected Response:** {visible UI state change}
**Verify:** {assertion on visible output -- NOT code-level assertion, per TSPEC-02}
```

Step 4: Build Mapping Table (per D-10 explicit mapping)
- Generate a table with columns: Test Case ID | Codepoint Sequence | Expected Firing Order
- Map each test case to the codepoints it exercises
- Uses point_id terminology from data-model.md

Step 5: Attach Probe References (per D-05 segment 5)
- For each test case category (button click, form submit, API call, state change), link the relevant probe snippet
- Format: `plugins/codepoint/references/test-probes.md#{category-anchor}`

Step 6: Output Test Plan (per D-09 storage location)
- Write the completed test plan to `.codepoints/test-plans/{flow_id}.md`
- Present to developer for review
- Include explicit user review gate: "Review the generated test plan. Adjust test cases or mapping as needed."

**Output Template section:** Show the exact 6-segment markdown template (per D-05):
- Segment 1: Title & Metadata (flow name, flow_id, collection_id, author, date)
- Segment 2: Preconditions (system state, data setup, feature flags)
- Segment 3: Test Case List (using TC-{N} format with Action/Expected Response/Verify)
- Segment 4: Codepoint Mapping Table (Test Case ID | Codepoint Sequence | Expected Firing Order)
- Segment 5: Probe Snippet References (links to test-probes.md sections)
- Segment 6: Notes & Deviations (optional, known gaps)

Provide a concrete example output template for `flow-user-login` showing all 6 segments filled in. The flow-user-login example MUST include all three test case categories explicitly:
- At least 1 normal test case (e.g., "User submits valid credentials")
- At least 1 boundary test case (e.g., "User submits empty form — all fields blank")
- At least 1 failure test case (e.g., "User submits invalid email format")
Each in the Action → Expected Response → Verify format.

**Integration section:** Explain where this fits:
1. After `/codepoint-scan` or `/codepoint-plan` produces flow definitions
2. Before `/codepoint-implement` inserts probes
3. Test plan stored in `.codepoints/test-plans/` alongside flows/ and points/

If the 4 base probe patterns (button click, form submit, API call, state change) don't cover a developer's testing scenario, direct them to consult `frontend.md` for framework-specific probe patterns, or generate a custom probe based on their description.

**Title:** `# Test Probe Code Snippets`

**Intro section:** Brief explanation -- these snippets are designed for frontend test verification. Copy the relevant pattern, replace placeholders (`{flow}`, `{name}`) with actual IDs from your `.codepoints/` definitions. All snippets use `pointWithMeta` from the base library (`@/lib/codepoint`).

**Category 1: Button Click** (anchor: `#button-click`)
- Basic variant: `pointWithMeta('cp-{flow}-button-click', { point_id, flow_id, button_id, timestamp })`
- Error handling variant: wraps the basic call in a try/catch, adds error metadata
- Each variant includes a complete code block with placeholder values and inline comments

**Category 2: Form Submit** (anchor: `#form-submit`)
- Basic variant: `pointWithMeta('cp-{flow}-form-submit', { point_id, flow_id, form_data_keys, validation_result })`
- Error handling variant: captures validation errors and submission failures
- Note: `form_data_keys` uses `Object.keys(formData)` to log field names without logging sensitive values

**Category 3: API Call** (anchor: `#api-call`)
- Basic variant: `pointWithMeta('cp-{flow}-api-call', { point_id, flow_id, endpoint, method, status })`
- Error handling variant: captures network errors and non-2xx responses with error message
- Note: Includes both start and end probe calls (before fetch and after response)

**Category 4: State Change** (anchor: `#state-change`)
- Basic variant: `pointWithMeta('cp-{flow}-state-change', { point_id, flow_id, from_state, to_state })`
- Error handling variant: captures unexpected state transitions

**Usage Notes section:**
- All snippets use pure JS/TS with no framework dependencies (per D-08)
- Import path: `import { pointWithMeta } from '@/lib/codepoint'` -- adjust to match project setup
- Replace `{flow}` with actual flow ID (e.g., `user-login`), `{name}` with descriptive name
- Each probe outputs JSON matching the V2 format from data-model.md
- For framework-specific patterns (React, Vue), see `plugins/codepoint/references/frontend.md`
