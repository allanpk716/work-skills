## 文档结构与内容要求

### 一、审查背景与方法

说明：
- 审查目的：对照原作者方法论（全局思维、集合论、密度校验）评估当前 Codepoint V2 技能设计
- 审查范围：scan/plan/implement 三阶段 SKILL.md，数据模型定义（data-model.md），主技能定义（codepoint/SKILL.md）
- 审查标尺：方法论原文 2.1-2.8 节，重点 2.2（谁来埋）、2.3（密度控制）、2.7（集合论）
- 审查方法：逐条比对原作者原则与当前设计实现，引用 E2E 测试产出作为实证

### 二、原作者方法论核心原则摘要

从 2026-04-17-methodology.md 和 2026-04-19-global-thinking.md 提炼以下核心原则作为比对基准：

1. **埋点决策者原则**（2.2 节）— 必须由深度理解系统的人手动埋设，AI 不能替代
2. **全局思维原则**（2.2 节）— 理解系统核心执行链路，不是逐个埋点
3. **集合论原则**（2.7 节）— 代码点组成集合，提示词+集合->LLM 筛选子集->解决问题
4. **密度校验原则**（2.3 节）— 堆栈交集太大=太密，完全没交集=太少
5. **按项目类型差异化原则**（2.3 节）— Tomcat ~20 vs 数据库 ~200+
6. **一次性埋设原则**（2.2 节）— 一次性手动埋设，不是循环迭代

每条原则必须引用方法论原文的原话（使用中文引用格式：> 原文内容），标注具体章节号。

### 三、设计偏差清单

偏差条目清单（per D-01, D-02, D-04, D-05, D-07, D-08）：

1. **CP-01: scan Phase 2 逐文件追踪 vs 全局思维**（D-01, D-02）
   - 引用方法论 2.2 节"一定要有全局思维"
   - 引用 scan SKILL.md Phase 2 Steps 第 2 步"Read the entry point handler/controller function, Trace function calls through service layer, repository layer, external calls"
   - 引用 gojs-calculator index.json 的 scan_notes: "Scan performed by analyzing source code" 佐证实际行为
   - 具体改进步骤：
     1. scan Phase 2 从"逐文件追踪"改为"链路导向"（per D-02）
     2. 只追踪入口→出口的完整调用链（如 HTTP handler -> service -> repo -> DB）
     3. 在调用链上只标记 4 类关键位置：模块边界、状态变更点、并发点、错误路径
     4. 不分析中间函数的内部逻辑
     5. 输出为 3-5 条核心链路图（而非完整的文件级报告）

2. **CP-02: Plan 阶段"新功能规划"定位 vs 集合构建**（D-04）
   - 引用方法论 2.7 节"所有的代码点也组成一个集合，只要找到一个代码点的子集就能解决各种问题"
   - 引用 plan SKILL.md "Plan code points for a new feature being developed"
   - 具体改进步骤：
     1. Plan 阶段重新定位为"集合构建"而非"新功能埋点规划"
     2. 基于 scan 的全局理解，识别 3-5 条核心执行链路
     3. 确定集合中每个代码点的位置和类型
     4. 确保代码点集合覆盖系统的所有关键执行路径
     5. Plan 的产出不是"探针方案"而是"代码点集合定义"

3. **CP-03: Implement 阶段 TDD 循环 vs 一次性埋设**（D-05）
   - 引用方法论 2.2 节和 3.2 节流程图（人工埋点是一次性的，不是迭代循环）
   - 引用 implement SKILL.md "TDD-style three-phase loop: Red → Green → Verify"
   - 具体改进步骤：
     1. Implement 阶段从 TDD 三步循环（Red/Green/Verify）改为"一次性埋设"
     2. 保留轻量验收环节（确认探针能触发、输出格式正确），去掉复杂的测试用例生成和 Verify 阶段
     3. 去掉"Boundary Condition Tests"和"Failure Mode Tests"的自动生成
     4. 保留 Phase 1（确认方案）和 Phase 2（插入探针），简化为两步

4. **CP-04: 统一密度阈值 vs 按项目类型差异化**（D-07）
   - 引用方法论 2.3 节"像 tomcat...埋 20 几个代码点就够了。数据库就复杂一些，代码点估计得有 200+ 个"
   - 引用 data-model.md "Target range: 20%-60% overlap between adjacent points"
   - 引用 go-calculator 的 12 个 codepoints 和 gojs-calculator 的 18 个 codepoints 作为小型项目参考
   - 具体改进步骤：
     1. 密度目标从统一 20%-60% overlap 改为按项目类型分级：
        - 简单应用（CRUD/Web 服务）：~10-30 个代码点，overlap 目标 30-60%
        - 中型系统（业务逻辑复杂）：~50-100 个代码点，overlap 目标 20-50%
        - 复杂系统（数据库/分布式）：~100-200+ 个代码点，overlap 目标 15-40%
     2. 在 scan 阶段根据项目规模自动推荐密度等级
     3. 将分级标准写入 data-model.md 的 Density Validation 章节

5. **CP-05: 密度校验仅概念描述 vs 需自动化执行**（D-08）
   - 引用方法论 2.3 节"如果两个代码点产生的堆栈交集很大...完全没交集说明埋得太少"
   - 引用 methodology.md 4.1 节的 AnalyzeOverlap 函数实现
   - 引用 scan SKILL.md Density Validation 章节 — 只有描述没有实际执行步骤
   - 具体改进步骤：
     1. 在 scan 阶段输出代码点后，自动执行密度校验（不需要等到 implement）
     2. 计算方法：使用 methodology.md 4.1 节的 AnalyzeOverlap 算法（堆栈帧集合交集 / 帧总数）
     3. 触发时机：scan Phase 2 完成代码点放置后，立即运行校验
     4. 输出校验报告到 `.codepoints/verification/density-report.md`
     5. 对每个 flow 中相邻代码点计算 overlap 值，标记超出范围的点

### 四、合理偏离说明

对以下合理偏离逐条说明为什么是工具化/Skill 化的必然调整：

1. **RD-01: AI 辅助 scan vs 人手动埋点**
   - 引用方法论 2.2 节"ai 都知道去哪里埋代码点了那还需要代码点干嘛！"
   - 说明为何合理：技能定位是"辅助工具"而非"替代决策"——AI 快速梳理代码结构，提出候选位置，scan Phase 1 有 User Confirmation 环节保留人的决策权
   - 引用 gojs-calculator index.json 的 cross_language_connections 说明 AI 辅助跨语言链路识别的价值

2. **RD-02: 三步流程（scan/plan/implement）vs 原作者两步（人埋设+AI 筛选）**
   - 说明为何合理：原作者描述的是理想流程，工具化需要将"人的理解"拆解为可执行的步骤（per D-03 保留三步结构）
   - scan 辅助理解，plan 构建集合，implement 执行埋设——三步对应三个明确的工具化操作

3. **RD-03: 三层模型（Collection/Flow/Point）的 Flow 有序性 vs 集合论的无序性**
   - 引用 RESEARCH.md Open Questions 中关于 Flow 的"有序组合"与集合论"子集"的关系
   - 说明为何合理：Flow 定义执行路径的顺序是实用性需要（帮助理解代码执行顺序），与集合论子集概念兼容——Flow 既是子集又有执行顺序的额外属性

### 六、与 E2E 测试结果的交叉验证

引用 2 个以上 E2E 测试项目的具体数据：

1. **go-calculator**（单语言项目）:
   - 12 个 codepoints, 3 flows, 1 collection
   - 4 个共享 codepoints (cp-calc-parse/validate/compute/format) 在所有 3 条链路中被复用
   - 分析：共享 codepoints 的存在符合集合论原则（"找一个代码点的子集就能解决各种问题"）

2. **gojs-calculator**（跨语言项目）:
   - 18 个 codepoints, 3 flows, 跨语言连接

   - 分析：跨语言链路识别是 scan 的真正价值所在，但当前 Phase 2 的逐文件追踪方式效率偏低

3. **交叉验证结论**：
   - 三层模型在 E2E 测试中运作良好（验证 D-06）
   - 共享 codepoints 的复用模式符合集合论原则
   - scan 的实际产出（index.json）质量好，但产出方式（逐文件追踪）可以优化为链路导向
