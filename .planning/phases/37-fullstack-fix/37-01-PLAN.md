- id: DEV-FS-01
- category: template

- description: "golang.md collector template (Frontend Collector section) is a simplified version that lacks sync.Mutex for thread safety, flow_id routing to per-flow log files, and JSON output for meta-bearing entries. The actual gojs-calculator project required an enhanced collector with all three features to support cross-language flow_id correlation."
- steps_to_reproduce: "1. Copy collector code from golang.md Frontend Collector section to new project\n2. Start Go server with concurrent browser requests\n3. Observe: no mutex protecting file writes, no flow_id-based file routing, meta (containing flow_id) is silently discarded\n4. Cross-language flow_id correlation fails"
- expected: "Collector has sync.Mutex for concurrent writes, routes entries with flow_id in meta to per-flow files (cp-ts-{flow}-{timestamp}.log), writes JSON for meta-bearing entries"
- actual: "Collector has no mutex, no flow routing, meta discarded -- only plain text [CODEPOINT] output to single file"
- root_cause: "Single-language E2E testing (Phase 32/33) does not exercise cross-language collector integration paths. The collector was tested in single-language mode where flow_id routing and meta serialization are not needed. Only fullstack testing (Phase 35) revealed that the collector must handle meta-bearing entries from frontend probes."
- status: fixed (in gojs-calculator project code; template NOT yet updated)

DEV-FS-03:

- category: template

- root_cause: "E2E testing was done on Windows but the SPA fallback code path was not exercised in single-language phases. Only fullstack testing with go:embed frontend serving (Phase 35 Plan 04 browser automation) triggered the Windows-specific path separator issue. Unit tests mock the file system and never hit embed.FS.Open with real paths."
- status: fixed (in gojs-calculator main.go using path.Clean; template example NOT yet updated)
