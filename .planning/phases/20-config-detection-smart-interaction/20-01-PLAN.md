<objective>
Enhance the Pushover configurator to detect credentials persisted via setx by reading the Windows registry in addition to process.env, and add per-item Confirm prompts allowing users to keep or re-enter each credential independently.

Purpose: Solves the detection blind spot where setx-persisted credentials are invisible to process.env until terminal restart (D-01). Enables granular per-item skip/update interaction (D-02, D-03).

Output: Enhanced pushover.js with registry detection, new i18n keys, updated tests.
</objective>

       - Add per-item detection with 4 cases:

       **Case A: Both token AND user exist** (per D-02, D-03):
       ```
       Display: "Pushover credentials already configured"
       Display: "  Token: {token.substring(0,8)}..."  (per D-04)
       Display: "  User: {user.substring(0,8)}..."    (per D-04)
       Confirm: t('pushover.keepConfig') with initial: true (per D-03, default Y = keep both)
       If user keeps: return { status: 'configured', details: 'already set' }
       If user doesn't keep: fall through to the "ask if wants to configure" Confirm, then full input+validation flow
       ```

       **Case B: Only token exists** (per D-02):
       ```
       Display: "Pushover credentials partially configured"
       Display: "  Token: {token.substring(0,8)}..."
       Confirm: t('pushover.keepToken', { value: token.substring(0,8) + '...' }) with initial: true
       If kept: save kept token value to variable `finalToken`
       If not kept: prompt Input for new token, save to `finalToken`
       Then prompt Input for user (no Confirm, it's missing)
       Validate finalToken + finalUser together through existing validation flow
       ```

       **Case C: Only user exists** (per D-02):
       ```
       Display: "Pushover credentials partially configured"
       Display: "  User: {user.substring(0,8)}..."
       Confirm: t('pushover.keepUser', { value: user.substring(0,8) + '...' }) with initial: true
       If kept: save kept user value to variable `finalUser`
       If not kept: prompt Input for new user, save to `finalUser`
       Then prompt Input for token (no Confirm, it's missing)
       Validate finalToken + finalUser together through existing validation flow
       ```

       **Case D: Neither exists** -- keep existing behavior (the "Ask if user wants to configure" Confirm + full input flow)

       IMPORTANT implementation details:
       - For cases B and C: After determining kept/new values, validate them with `validatePushoverCredentials()`, save with `setEnvVariable()`, update `process.env` -- reuse the EXISTING validation+save logic (per RESEARCH "Don't Hand-Roll")
       - For case A when user chooses to re-enter: fall through to the normal "Configure Pushover?" prompt, then full retry loop
       - Do NOT change the setx persistence mechanism or the validation logic -- only change detection and interaction
       - Keep the existing retry loop structure for cases where user enters new credentials
       - The `enquirer` Confirm `initial: true` means default Y = keep current config (per D-03)

    4. Export `detectPushoverFull` in module.exports (keep all existing exports, add the new one)
