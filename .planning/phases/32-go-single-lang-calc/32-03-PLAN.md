## 1.1 Run /codepoint:plan on existing codebase

In the Go calculator project directory `tmp/go-calculator/`, trigger `/codepoint:plan`.

Provide this as the plan input — a refinement request for existing flows, NOT a new feature:

```
Plan Request: Codepoint probe placement for existing calculator flows

Goal: Optimize probe placement across the 3 existing business flows to maximize
stack trace differentiation and debugging value.

Existing flows (from scan):
1. REST API Calculate: POST /api/calculate -> calculator.Evaluate() -> Parse -> Validate -> Compute -> Format -> JSON response
2. Batch Processing: CLI --batch -> calculator.Evaluate() -> Parse -> Validate -> Compute -> Format -> WriteOutput
3. History Query: GET /api/history/{id} -> Store.Lookup -> calculator.Evaluate() -> Parse -> Validate -> Compute -> Format -> HistoryResponse

Key architecture:
- calculator.Evaluate(ctx, expr) is the shared entry point wrapping Parse->Validate->Compute->Format
- Each flow enters through a different handler/function
- HandleHistoryGet RECOMPUTES stored expressions through the full shared pipeline

Probe placement goals:
1. Each flow needs at least 1 entry probe (at handler/function entry)
2. Shared path needs probes that capture different stack traces per flow
3. flow_id must be propagated via context.Context from flow entry to shared path probes
4. At least 1 error-type probe for error paths (division by zero, invalid syntax)
5. Target probe density: 20-60% stack overlap between adjacent probes
```

## 1.3 Verify probe placement quality

Check the plan output against these quality criteria:

1. **Probe location rationality**: Every probe should be at a meaningful location:
   - `entry` type: at HTTP handler function entry or batch processor entry
   - `boundary` type: at cross-module call boundaries (e.g., api -> calculator, batch -> calculator)
   - `state-change` type: at state transitions (e.g., before/after computation, result formatting)
   - `error` type: at error handling paths (e.g., division by zero, parse failures)
   - **NOT at**: package-level variable declarations, init functions, pure helper utilities

2. **Shared code point reuse**: Plan should reference the calculator.go functions (Parse/Validate/Compute/Format or Evaluate) as shared probes across flows. If plan creates entirely new point IDs for each flow without reuse, note this as a finding.

3. **Test case outline**: Each flow should include normal/boundary/failure test scenarios in the plan.

4. **flow_id propagation**: Plan should indicate how flow_id reaches shared path probes. Recommended: context.Context values set at flow entry, read at probe sites. If plan uses a different mechanism (function parameter, goroutine-local), that is also acceptable.

5. **Code point naming**: Follows `cp-{module}-{action}-{position}` convention. Variations accepted since scan naming may vary.

## 1.4 Verify plan output is implement-ready

Plan output must be consumable by the implement skill. Verify:

1. `.codepoints/index.json` is updated with the plan results
2. Flow documents contain complete sequences with point IDs, types, and locations
3. Point documents contain: id, location (file:line), type, description
4. The structure matches what `/codepoint:implement` expects as input

## 1.5 Record findings (do NOT fix)

If the plan skill has issues, record but do not fix:
- Probes placed at non-critical locations
- Failure to reuse existing shared code points
- Missing test case outlines
- Inconsistent naming
- No flow_id propagation strategy
