### v2.0 - 前端自动化测试体系 (In Progress)

**Milestone Goal:** 建立前端开发的测试规划+埋点体系，让每个新功能都有可验证的测试流程

- [ ] **Phase 41: Test Specification Foundation** - 测试规划模板与探针代码片段
- [ ] **Phase 42: Codepoint Data Integration** - .codepoints/ 结构读取与测试用例关联
- [ ] **Phase 43: Instrumentation-First Planning** - 埋点优先规划与探针放置指导
- [ ] **Phase 44: Verification Automation** - 探针验证、覆盖度分析与流程映射
- [ ] **Phase 45: Skill UX Orchestration** - 特征集成的用户交互体验
- [ ] **Phase 46: Progressive Validation** - 渐进式跨项目验证

## Phase Details

### Phase 41: Test Specification Foundation
**Goal**: Developers have a standardized format for planning frontend test flows with ready-to-use probe code snippets
**Depends on**: Nothing (first v2.0 phase, builds on v1.9.2 foundation)
**Requirements**: TSPEC-01, TSPEC-02, TSPEC-03
**Success Criteria** (what must be TRUE):
  1. Developer can use a test plan template with sections for preconditions, click->response->verify steps, and codepoint flow mapping
  2. Developer can write test cases in a standardized format specifying: action, expected UI state change, and assertion on visible output
  3. Developer can copy pre-built probe code snippets for common frontend patterns (button click, form submit, API call, state change) into their codebase
  4. Template output aligns with existing .codepoints/ directory conventions (uses flow_id, point_id terminology)
**Plans**: TBD

### Phase 42: Codepoint Data Integration
**Goal**: The skill reads existing .codepoints/ instrumentation data and maps test cases to instrumented flows
**Depends on**: Phase 41
**Requirements**: CPT-01, CPT-02
**Success Criteria** (what must be TRUE):
  1. Skill reads and parses .codepoints/index.json to extract flow definitions, point definitions, and cross-language connections
  2. Developer can query "which test cases exercise this codepoint flow?" and get a mapped answer
  3. Test plan output is organized by collection structure (not individual features), matching the Codepoint V2.1 direction
**Plans**: TBD

### Phase 43: Instrumentation-First Planning
**Goal**: Developers plan probe placement during feature design, not after implementation
**Depends on**: Phase 42
**Requirements**: INST-01, INST-02, INST-03
**Success Criteria** (what must be TRUE):
  1. /codepoint-plan skill includes frontend-specific probe placement guidance (event handlers, data fetch hooks, state transitions) in its output
  2. Developer can plan instrumentation at the same time as feature design -- probe placement guidance is generated alongside the feature plan, not as a separate step
  3. Developer can reference curated probe placement patterns for React hooks, form handling, async data loading, error boundaries, and routing transitions
**Plans**: TBD

### Phase 44: Verification Automation
**Goal**: Developers can verify probe activation, map test cases to flows, and identify coverage gaps
**Depends on**: Phase 43
**Requirements**: VERF-01, VERF-02, VERF-03
**Success Criteria** (what must be TRUE):
  1. Developer can verify probes fire correctly through lightweight acceptance -- confirm probe activation, output format, and firing order
  2. Skill automatically maps each test case to the codepoint flow it exercises, enabling "which test cases cover this codepoint?" queries
  3. Developer can analyze test plan coverage density -- identify flows with no test cases and flows with redundant tests
**Plans**: TBD

### Phase 45: Skill UX Orchestration
**Goal**: Developers can generate full test plans from feature specs and span cross-stack flows
**Depends on**: Phase 44
**Requirements**: UX-01, UX-02
**Success Criteria** (what must be TRUE):
  1. Developer provides a feature spec or describes a feature, and the skill generates a structured test plan reading from .codepoints/index.json
  2. Developer can plan test flows that span frontend->API->backend using cross_language_connections from index.json
**Plans**: TBD

### Phase 46: Progressive Validation
**Goal**: The test plan template works across Go+JS and Python+TS projects with coverage comparison
**Depends on**: Phase 45
**Requirements**: UX-03
**Success Criteria** (what must be TRUE):
  1. Developer can run the same test plan template against Go+JS and Python+TS projects
  2. Developer gets a comparison report showing coverage gaps between projects
  3. Test plans for gojs-calculator (18 codepoints, 3 flows) validate end-to-end as pilot proof
**Plans**: TBD
